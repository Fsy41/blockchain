## 一、强匹配项：主动展开，用真实代码细节建立可信度

这部分是面试官最想听、也是你最该主动展开的。每个点都附了"可追问的代码细节"，证明你真的做过。

### 1. Agent 编排引擎（命中 JD 职责1、4）—— 最核心，必讲

**怎么讲：**
"我参与开发了一套基于 LangGraph StateGraph 的 Agent 编排引擎，核心是一个 Plan+Executor 架构，完整节点流程是：`prepare → 意图解析 → 条件路由 → 直接响应/任务规划 → 执行器循环 → 验证器 → 响应`。"

**展开细节（面试官追问时抛出）：**

- **意图分流**：LLM Function Calling 输出意图类型 + 置信度，**置信度 < 0.7 强制走规划**（保守策略），实现 type=1（直接响应）/ type=2（进入规划）动态分流。
- **任务规划**：一次性生成结构化 ExecutionPlan，每个 ExecutionStep 含 `step_id / tool_name / depends_on 依赖链 / allow_failure 降级 / retry_count / expected_output_description`。
- **ReAct 式执行**：执行器流式生成 ToolCall → `tool_choice` 强制指定当前步骤工具 → ToolNode 执行 → 流式回传前端，配合**指数退避重试**（1s→2s→4s，最多 3 次）。
- **验证与自修正**：Verifier 节点每步验证，支持 `continue / self_correct / complete` 三态决策，自修正上限可配（0-5 次），关键步骤失败触发早期终止与降级执行（Quick Wins）。
- **多 Agent 编排**：父图工厂通过 `add_conditional_edges` 支持**顺序、并行、条件分支、循环**工作流模式与单/多 Agent 混合编排，Plan 与 React 策略可由租户配置切换。

**量化成果（必说）：**

> "Plan+Executor 模式相较 Orchestrator 模式，**响应时间降低 50%、token 消耗降低 60%**，适用于 70% 的简单/中等复杂度任务。"

三段式话术：

> **② 用真实案例证明迁移成本低：**

**如果被深挖座舱具体业务（车控指令、语音唤醒等）：**

## 三、面试官高频追问 + 标准回答

### Q1："你用 LangGraph 和自己手写编排引擎有什么区别？为什么要用 LangGraph？"

**回答框架：**

> "我用 LangGraph 是因为它提供了 StateGraph 状态机抽象、条件边、Checkpointer 持久化这些基础设施，避免重复造轮子。但 LangGraph 是**通用图编排工具**，不是 Agent 框架——我在它之上构建了**面向业务的抽象层**：意图解析、任务规划、工具注册、验证自修正、多租户配置，这些 LangGraph 不提供。可以说我用 LangGraph 做'底层执行引擎'，自己做'上层 Agent 框架'。"

### Q2："Plan+Executor 和 ReAct 的区别？什么时候用哪个？"

**回答框架：**

> "ReAct 是'想一步做一步'，每步都让 LLM 决策，灵活但 token 消耗高、轮次多；Plan+Executor 是'先规划再执行'，一次生成完整计划再逐步执行，**token 消耗降低 60%、响应时间降低 50%**，适合任务结构清晰的场景。我的框架支持租户配置切换两种策略，70% 的简单/中等任务用 Plan+Executor 更优，复杂探索性任务用 ReAct 更灵活。"

### Q3："你的工具框架怎么保证工具调用的可靠性？"

**回答框架：**

> "多层保障：① 指数退避重试（1s→2s→4s，最多 3 次）；② 验证器三态决策（continue/self_correct/complete）；③ 关键步骤失败早期终止（Quick Wins）；④ allow_failure 降级标记，允许非关键步骤失败继续执行；⑤ 三级超时控制（首字/块间隔/总时长）。"

### Q4："多租户怎么隔离的？"

**回答框架：**

> "三个维度：① 数据库 Schema 隔离；② Redis 用 `tenant_id:*` key 前缀隔离；③ 对象存储按 Bucket/Container 隔离。配置层面，工具/Prompt/MCP 配置都按 tenant_id 加载，核心层零业务逻辑。"

### Q5："记忆系统怎么做的？怎么避免上下文爆炸？"

**回答框架：**

> "四种摘要策略可插拔：基于 token 阈值、基于轮次阈值、基于最近用户轮、全局简单摘要。超过阈值自动触发摘要压缩，压缩后注入 prompt 的 `<<PREV_SUMMARY>>` 占位符。持久化用 PostgresSaver Checkpointer，我参与了表分片策略设计（按年月+取模分表）支撑大规模会话。"

### Q6："SKILL.md 这种自然语言指令，LLM 不遵守怎么办？"（针对 Skill.Agent 项目）

**回答框架：**

> "多层约束：① 输出契约固定为裸 JSON（indicatorList + nextTasks），配合'输出前自检'清单；② 不含 JSON 时自动补发一轮重试，解析失败降级为默认值；③ 工具返回字段裁剪控制 token；④ 评测管线（130 条测试集）持续验证。当然完全约束 LLM 是难题，所以我在 SKILL.md 里把约束写得非常具体（如'第 3 轮起禁止重搜'），用硬规则而非软建议。"
