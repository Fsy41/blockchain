### 项目一：企业级多租户通用 Agent 开发框架　（参与核心模块研发）

**角色**：平台研发（参与编排引擎、LLM 适配层、工具集成框架、记忆系统、可观测性等模块开发）

**项目简介**：面向多业务方的 Agent 开发平台，业务方以"插件租户"方式接入，复用平台的意图解析、任务规划、工具集成、状态管理、记忆等基础设施。框架代码约 6 万行、313 个模块，核心层零业务逻辑。

#### 1. Agent 编排引擎（Plan+Executor 架构）　（对应 JD 职责 1、4）

参与开发基于 LangGraph StateGraph 的 Agent 编排引擎。**一次对话的完整流程是这样的：**

```
用户提问
  → 准备阶段（收集会话历史、加载工具、组装 Prompt）
  → 意图解析（LLM 判断：简单闲聊就直接答，复杂任务就进入规划）
  → 任务规划（LLM 把任务拆成有序步骤列表）
  → 执行器循环（按步骤逐个调用工具，每步可重试）
  → 验证器（检查结果是否合理，不合理可触发自修正重做）
  → 响应（LLM 组织成自然语言回答用户）
```

核心设计点：

- **意图分流**：先判断问题复杂度，简单问题直接回答省时间，复杂问题才走完整规划流程。
- **先规划后执行**：一次性生成完整计划再逐步执行，而非走一步看一步，减少 LLM 决策轮次。
- **验证与自修正**：每步执行后校验结果，失败可重试或降级跳过，关键步骤失败提前终止。
- **多 Agent 编排**：通过状态机的条件分支，支持多个 Agent 按顺序、并行、条件分支等方式协作。

> **成果**：Plan+Executor 模式相比逐步决策模式，**响应时间降低约 50%、token 消耗降低约 60%**。

#### 2. 插件化工具集成框架　（对应 JD 职责 2）

参与设计工具注册与调用框架，核心思路是**让新工具的接入零硬编码**：

- **装饰器声明式接入**：工具开发者用装饰器（`@pre_process` / `@post_process`）声明工具调用前后的处理逻辑，框架自动扫描注册，无需手写注册代码。
- **多源工具统一管理**：本地工具、MCP 远程工具、A2A Agent 工具三类来源统一注册到一个"工具目录"，查找优先级可配置。
- **配置优先级**：租户配置 > 装饰器配置 > 默认配置，灵活适配不同场景。

#### 3. 统一 LLM 适配层　（对应 JD 任职要求 3）

基于 LiteLLM 封装统一调用层，适配 10+ 大模型供应商：

- **配置化切换**：换模型只改配置，代码不动——这是"多推理后端无缝切换"的基础。
- **动态模型选择**：不同环节可用不同模型（如规划用强模型、执行用快模型）。
- **故障转移**：主模型不可用时自动切换备选模型。

#### 4. 记忆与上下文管理　（对应 JD 职责 1）

- **多策略记忆摘要**：对话太长时自动把历史压缩成摘要，避免超出上下文窗口。支持基于 token 数、轮次数等多种触发策略。
- **会话持久化**：基于 PostgreSQL 的 Checkpointer，支持跨会话状态恢复。

#### 5. 可观测性与性能优化　（对应 JD 职责 3、5）

- **全链路追踪**：集成 Langfuse 记录 LLM 每轮调用、OpenTelemetry 做分布式追踪、自研节点埋点上报耗时与成本。
- **流式输出**：全链路 SSE 流式，用户不用等全部生成完才看到结果。
- **资源优化**：数据库连接池复用、工具元数据缓存、Prompt Cache 降低 token 成本。

### 项目三：EDB.Skill.Agent —— Skill 驱动的 Agent 框架　（独立主导，从 0 到 1）

**角色**：框架设计与开发（独立完成）

**项目简介**：这是一次架构迁移——把原有的"10 节点硬编码状态机"智能搜索 Agent，重构为**加载 Skill、按 Skill 指令自主规划的 ReAct Agent**。编排逻辑从"写死在代码里"迁移到"一份可维护的自然语言指令文档(SKILL.md)"，针对搜索准确率做了专项优化。

#### 1. 架构迁移：从代码状态机到 Skill 驱动　（对应 JD 职责 1）

**痛点**：旧方案是 10 个节点的固定状态机，改任何搜索逻辑都要改 Python 代码、改图结构，扩展成本高。

**方案**：用一个通用 ReAct Agent + 可加载的 Skill 文件替代固定状态机。

```
旧：用户问句 → 固定状态机(10节点) → 输出
新：用户问句 → ReAct Agent → 读SKILL.md指令 → 自主规划 + 调工具 → 输出
```

新增业务场景 = 写一份新的 SKILL.md，不改框架代码。

#### 2. Skill 加载系统设计　（对应 JD 职责 2）

独立设计并实现完整的 Skill 加载框架：

- **Skill 文件规范**：定义 `SKILL.md` + YAML frontmatter 格式，用 frontmatter 声明技能名称、用途、可用工具，正文写自然语言工作流指令。
- **加载协议抽象**：定义统一的加载接口（SkillLoaderProtocol），加载来源可插拔——已实现本地加载，预留远程加载扩展点。
- **SkillManager**：启动时加载所有技能元数据、生成技能清单、收集技能允许使用的工具。
- **两阶段加载**：启动时只加载技能清单（轻量元数据），运行时按需加载技能正文，避免撑爆上下文窗口。
- **预注入优化**：技能名称已知时，请求时直接预加载技能正文注入，省掉一轮 LLM 往返。

#### 3. 检索编排设计（提升搜索准确率）　（对应 JD 职责 1）

在 SKILL.md 中用自然语言定义完整的查询编排策略，提升检索准确率：

- **Query 标准化与拆解**：把口语化问句拆成精准的指标查询（如"煤油比"展开为"煤炭原油价格比"）。
- **并发搜索**：独立的查询组在同一轮并发发起，降低多指标查询延迟。
- **两轮搜索约束**：第一轮并发搜全部；仅对失败组基于原意改写重搜一次；第三轮起禁止，控制成本。
- **匹配校验与过滤**：对返回结果逐项校验（经济体/频率/口径/单位），剔除不相关候选。
- **固定输出契约**：只输出结构化 JSON，禁止臆造数据。

#### 4. 可观测性与评测　（对应 JD 职责 3）

- **ReActTracer**：基于 LangChain Callbacks 实现，记录每轮推理链、工具调用入参出参，支持 SSE 实时推流。
- **评测管线**：搭建批量评测体系（130 条问句测试集 + 跑批脚本 + 多版本对比），支撑迭代调优。
