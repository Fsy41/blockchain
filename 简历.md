### 项目一：企业级多租户通用 Agent 开发框架 Wind.Agent.Platform　（参与核心模块研发）

**角色**：平台研发（参与编排引擎、LLM 适配层、记忆系统、工具集成框架、可观测性等核心模块开发，并在平台上落地 EDB 租户业务）

**项目简介**：面向多业务方的企业级多租户 Agent 平台，业务方以"插件租户"方式快速搭建专属 Agent，复用平台意图解析、任务规划、工具集成、状态管理、记忆、可观测性等基础设施。框架代码约 6 万行、313 个模块，核心层零业务逻辑。

#### 1. Plan+Executor 通用编排引擎　（对应 JD 职责 1、4：意图解析、任务规划、多 Agent 协作）

参与开发基于 LangGraph StateGraph 的完整 Agent 编排引擎，节点流程为：
`prepare → 意图解析 → [条件路由] → 直接响应 / 任务规划 → 执行器(循环) → 验证器 → 响应`

- **意图解析与分流**：LLM Function Calling 输出意图类型 + 置信度，置信度 < 0.7 强制走规划（保守策略），实现 type=1（直接响应）/ type=2（进入规划）动态分流。
- **任务规划**：一次性生成结构化 ExecutionPlan（含 step_id / tool_name / depends_on 依赖链 / allow_failure 降级 / 重试次数 / 期望输出描述）。
- **ReAct 式执行**：执行器流式生成 ToolCall → `tool_choice` 强制指定当前步骤工具 → ToolNode 执行 → 流式回传前端，配合指数退避重试（1s→2s→4s，最多 3 次）。
- **验证与自修正**：Verifier 节点每步验证，支持 `continue / self_correct / complete` 三态决策，自修正上限可配（0-5 次），关键步骤失败触发早期终止与降级执行。
- **多 Agent 协作编排**：实现父图工厂，通过 `add_conditional_edges` 支持**顺序、并行、条件分支、循环**等工作流模式与单/多 Agent 混合编排，Plan 与 React 策略可由租户配置切换。

#### 2. 配置驱动 + 装饰器声明的工具集成框架　（对应 JD 职责 2：插件化工具集成、声明式/注解接入）

参与设计**策略模式 + 注册表 + 三级配置优先级**的工具处理体系：

- **声明式接入**：`@pre_process` / `@post_process` / `@a2a_tool` 装饰器声明工具的前后处理逻辑与 A2A Agent 工具注册，配合自动扫描机制，业务方零硬编码扩展。
- **三源统一注册**：统一注册 Local / MCP 远程 / A2A Agent 三类工具源，查找优先级可配（默认 local > mcp > a2a），同名本地编排器可覆盖远程工具。
- **三级配置优先级**：租户配置 > 装饰器配置 > 默认配置，灵活适配不同业务场景。

#### 3. 标准化大模型接入适配层　（对应 JD 任职要求 3：多推理后端无缝切换）

基于 LiteLLM 封装 ModelFactory + LLMCaller 统一治理层，适配 OpenAI / Anthropic / DeepSeek / 通义 / 豆包 / 智谱 GLM 等 10+ 供应商：

- **请求级动态模型选择**：可按节点 `{planner: "xxx", executor: "yyy"}` 指定不同模型；支持 model_seq 故障转移按序切换。
- **结构化输出双路径**：OpenAI 原生 response_format / LangChain function-calling（with_structured_output）自动选择。
- **Anthropic Prompt Cache 自动注入**，三级超时控制（首字 15s / 块间隔 30s / 总时长 300s），切换推理后端仅需改配置无需改代码。

#### 4. 多策略记忆与上下文管理　（对应 JD 职责 1：记忆机制、职责 5：上下文压缩）

- 参与开发可插拔的记忆摘要框架，实现 4 种 Summarization 策略（TokenBased / TurnBased / RecentUserTurnBased / SimpleGlobal）与 Compaction 策略，对长会话自动摘要归档。
- 参与自研异步 Checkpointer（PostgresSaver）并实现**表分片策略**（按年月 + 取模分表），支撑大规模会话持久化与跨会话状态恢复。

#### 5. Agent 调试与可观测性平台　（对应 JD 职责 3）

### 项目三：EDB.Skill.Agent —— Skill 驱动的 Agent 框架　（独立主导设计与实现，从 0 到 1）

**角色**：框架设计与开发（独立完成）

**项目简介**：这是一次 Agent 架构范式的迁移实践——将原有的"10 节点硬编码 LangGraph 状态机 + 提示词驱动工具"的智能搜索 Agent，重构为**加载 Skill、按 Skill 正文指令自主规划的 ReAct Agent**。编排逻辑从"写死在代码里"迁移到"自然语言指令(SKILL.md)"，针对智能搜索准确率做了专项优化。

#### 1. 架构范式迁移：从代码状态机到 Skill 驱动　（对应 JD 职责 1：整体架构）

**痛点**：原 EDB.Search.Agent 是 10 节点 LangGraph 状态机（prepare → intent → search → plan → execute → validate → retry → output），编排逻辑全部写死在代码和 prompt 模板里，每加一个新场景都要改图结构、加节点、改路由。

**方案**：用通用的 ReAct Agent + 可加载的 Skill 文件替代写死的状态机。

```
旧: 用户问句 → 状态机(10节点固定流程) → execute节点用prompt生成参数 + 手动invoke工具 → 输出
新: 用户问句 → ReAct Agent → 读SKILL.md指令 → 自主规划 + 原生Function Calling调工具 → 输出
```

编排逻辑（先做什么、再做什么、何时停止）从代码迁移到 SKILL.md 正文；新增业务场景 = 写一个新 SKILL.md，无需改任何框架代码。

#### 2. Skill 加载系统设计　（对应 JD 职责 2：插件化框架）

独立设计并实现完整的 Skill 加载框架：

- **Skill 文件规范**：定义 `SKILL.md` + YAML frontmatter 的技能描述格式，frontmatter 声明 `name`、`description`、`when_to_use`、`allowed-tools`、`model`，正文是自然语言工作流指令。
- **加载协议抽象 `SkillLoaderProtocol`**：定义 `load_skill_metas`、`load_skill_content`、`validate_config`、`provides_tool` 等统一接口，加载来源可插拔（已实现 LocalSkillLoader，预留 Remote/Filesystem 扩展点）。
- **SkillManager**：启动时加载所有技能元数据、生成技能清单(SKILLS_META)、创建 `get_skill` 工具（带 4 级模糊路径匹配：精确 → 后缀 → 文件名 → 打分）、收集 allowed_tools。
- **两阶段延迟加载**：启动时只注入技能清单（元数据，每个技能一句话），运行时按需加载技能正文，避免撑爆上下文窗口。
- **预注入优化**：当 skill 名称已知时，请求时直接将 SKILL.md 正文预注入为 SystemMessage，省掉"先调 get_skill 再干活"的一整轮 LLM 往返。

#### 3. Skill 正文设计与检索编排（提升准确率）　（对应 JD 职责 1：任务规划）

独立撰写 `economic-indicator-search` SKILL.md（200+ 行），将原状态机的全部业务逻辑翻译为自然语言工作流指令，针对检索准确率做了多层约束：

- **Query 标准化与语义拆解**：把口语化问句拆成高度相关的指标查询组（如"煤油比"展开为"煤炭原油价格比"，非机械处理），禁止 3+ 指标混入单个查询。
- **两轮并发搜索硬约束**：独立查询组必须在同一响应并发发起多个 tool_call；仅失败组基于原意改写（纠错称/转别名/官方名/收窄范围）重搜一次；**第 3 轮起硬性禁止**，控制 LLM 往返与成本。
- **匹配校验与过滤**：对返回候选逐项比对经济体/地区/频率/口径/单位/季调/覆盖范围，剔除多余/泛化/相邻主题候选。
- **量纲一致性检查 + 固定输出契约**：比值/占比/差值运算检查量纲，不一致写入 nextTasks 而非强行搜索；只输出裸 JSON（indicatorList + nextTasks），禁止臆造未确认指标。

#### 4. ReAct 全链路可观测　（对应 JD 职责 3）

独立实现 `ReActTracer`（基于 LangChain Callbacks）：

- **LLM 调用追踪**：每轮推理链(reasoning_content)、判断(content)、决策(调工具还是最终回答)。
- **工具执行追踪**：每个工具的开始/入参/结果/异常。
- **SSE 流式推送**：通过 `/search/stream` 端点实时推送执行过程给前端（thinking / tool_start / tool_end 等事件）。
- **输出兜底重试**：LLM 回复不含 JSON 时自动补发一轮"只输出 JSON"提示重试，解析失败降级为默认值。

#### 5. 评测管线　（对应"工程化思维"）

搭建批量评测管线（130 条问句测试集 + 跑批脚本 + 断点续跑 + 多版本对比实验），配套质量等级人工评级，支撑工具描述与 Skill 的迭代调优；另实现 LLM 代理可用性探测器，自动扫描多 Provider 配置逐个测试连通性与延迟。
