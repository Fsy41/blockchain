## 项目经历

### 项目一:Wind.Agent.Platform —— 多租户 Agent 开发平台(框架底座研发)

**角色**:框架研发工程师(负责平台底座、Agent 编排引擎、LLM 适配层、可观测体系)

**项目简介**:面向多业务方的企业级多租户 Agent 平台,业务方以插件方式接入,即可快速获得意图解析、任务规划、工具调用、记忆管理等完整 Agent 能力。平台已支撑 EDB(经济数据库)租户,提供智能检索、指标分析、图表生成、专家点评等能力。

#### 1. 多租户平台架构与插件化框架(对应 JD:职责 1、2 / 要求 4)

- 设计**租户插件契约**:每个租户以独立插件包形式接入,通过标准 `register()` 契约声明租户 ID、租户名、Graph 策略与 Agent 映射,框架自动发现并加载,实现**业务方零侵入接入**。
- 实现**租户级隔离**:工具(`plugins/<tenant>/tools/`)、Prompt、Agent 配置、MCP 连接器、Redis 缓存、MinIO 对象存储均按 `tenant_id` 隔离;通过 Schema 隔离 + 应用层校验保障多租户数据安全。
- 设计**工具注册中心 `ToolRegistry`**:统一管理「核心工具(目录自动发现 `ToolDiscovery`)+ 租户本地工具 + MCP 远程工具」三类来源,支持按名检索、按优先级(`local`/`mcp`)查找;工具按租户缓存,避免重复加载。

#### 2. 基于 LangGraph 的 Agent 编排与状态机引擎(对应 JD:职责 1、4)

- 基于 **LangGraph StateGraph** 设计智能检索 Agent 的图结构,落地**条件分支路由**:意图路由节点根据置信度将请求分流至「快速路径(直接检索)」或「完整路径(规划-执行-验证)」,兼顾低延迟与复杂任务完成率。
- 实现**计划-执行-验证-重试循环**:`plan → execute(可循环步进)→ validate → retry`,验证失败按置信度分级重试,并引入**全局时间预算控制**(`FULL_PATH_BUDGET_SECONDS`),超时自动降级输出,保障响应实时性。
- 设计**两级图编排(Parent Graph + Subgraph)**:Parent Graph 的 coordinator 节点负责上下文初始化与租户子图注入,子图继承 Context 与 Checkpointer,支持**状态持久化与断点续跑**。
- 抽象 **Runtime Context**:通过 `Context`(dataclass)+ `context_provider` 装饰器在图执行期间向所有节点/子图注入租户信息、工具注册表、模型配置、会话上下文,实现依赖注入与解耦。

#### 3. 状态管理与记忆机制(对应 JD:职责 1)

- 设计 **AgentCtx** 上下文管理类,统一管理全局上下文(`global_ctx`)、会话历史(`session_ctx`)、动态上下文(`dynamic_ctx`)、历史摘要(`previous_summary`)与轮次状态(`turns_state`)。
- 实现**记忆压缩**:将长会话历史摘要为 `previous_summary`,配合 `build_session_messages` 智能截断,控制上下文窗口占用,降低 token 成本。
- 实现 **Checkpointer 分表策略**:基于 UUIDv7 的时间分表(`TableShardingStrategy`),按年月对会话状态表分片,支撑长周期会话历史的持久化与高效检索。

#### 4. 大模型接入适配层(对应 JD:要求 3)

- 设计 **ModelFactory**,基于 **LiteLLM** 统一封装 10+ 推理后端(Anthropic / OpenAI / DeepSeek / Moonshot / Gemini / GLM / 通义 / 火山 / 自定义等),通过 `provider/model` 字符串解析 + 配置注入,业务层一行配置即可切换模型,实现**推理后端无缝切换**。
- 统一封装 `ChatLiteLLM`,抹平各厂商在流式输出、Function Calling、多模态上的差异;支持 `parallel_tool_calls`、`tool_choice` 等能力开关。
- 实现 **Prompt/Tools 级 Cache Control**:针对超长 system prompt 与工具描述,注入 `cache_control` 标记,复用 Prompt 前缀缓存,显著降低重复调用的 token 消耗与首 token 延迟。
- 接入各厂商**模型成本表**(`model_cost`),实现按 token 的精细化成本核算。

#### 5. 可观测性与调试平台(对应 JD:职责 3)

- 集成 **Langfuse** 全链路追踪:通过 LiteLLM 的 `success_callback` / `failure_callback` 注入,每次 LLM 调用自动上报 trace_id、session_id、turn_id、用户输入、模型响应、token 用量与成本,支持按会话/轮次回放与诊断。
- 设计**全链路标识体系**:`trace_id / thread_id / turn_id / master_thread_id` 贯穿 Parent Graph、子图、工具调用,任意节点行为可溯源。
- 实现 A2A 流式消息与 **EventHandler 事件路由**:将 Agent 的思考、工具调用、UI 组件渲染等事件统一编排为流式消息输出,支持组件级状态管理与前端实时渲染。

### 项目二:alice-chat(oh-my-winus)—— Agent Skill 技能系统(设计与研发)

**角色**:Skill 框架设计与开发(主导技能加载框架与技能创建 Agent)

**项目简介**:为对话式 Agent 平台新增**技能(Skill)系统**,使 Agent 能够动态加载、创建、更新技能(以 Markdown + Frontmatter 描述的可复用能力包),显著降低新能力扩展成本。

#### 1. Skill 加载框架设计(对应 JD:职责 2 插件化 / 要求 4)

- 设计**统一加载协议 `SkillLoaderProtocol`**(ABC),抽象 `load_skill_metas` / `load_skill_content` / `validate_config` / `provides_tool` 等接口,实现「**Local(本地技能库) / Filesystem(文件系统) / Remote(远程技能服务)**」三种加载器的可插拔切换。
- 实现 **Frontmatter 解析**:基于 YAML 解析技能文件的元数据(`name` / `description` / `when_to_use` / `allowed_tools` / `model`),并将技能清单聚合为 `SKILLS_META` 注入 system prompt,供模型按需调用。
- 设计 **SkillsMiddleware** 中间件:与 Agent 中间件链集成,在 `abefore_agent` 注入技能元数据、`abefore_model` 动态下发 `allowed_tools`,实现技能与工具的协同管控(支持 `ignore_tools` 过滤、子技能工具去重合并)。

#### 2. Create Skill Agent —— 技能自动创建(对应 JD:职责 4 多 Agent 编排)

- 基于 ReAct Agent 实现**技能自动创建/修改流程**:Agent 根据用户自然语言描述,自动调用 `generate_skill_content` 生成技能内容,经用户确认后通过 `upload_skill` / `update_skill` 完成持久化。
- 实现 **Human-in-the-loop 人工介入**:通过 `stop_by_tools`(调用 `skill_clarify_tool` 即暂停)与 `must_last_tool` 约束 ReAct 循环终止条件,结合独立 `human_node` 构成「生成 → 澄清确认 → 落库」的交互闭环。
- 设计**内外轨消息隔离**(`_make_hidden_tools_writer`):内部编排工具(如技能选择器)的调用事件被过滤,不透出至前端 UI,保持用户界面简洁,同时完整保留内部执行链路用于调试。
